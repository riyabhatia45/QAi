# Complete Code Walkthrough - Self-Healing Dashboard

The goal of this dashboard is:

1. Show latest test run health at a glance.
2. Show healing activity (selector heals, flow recoveries, failures).
3. Show local Git activity and working tree changes.
4. Show selector memory entries learned by the framework.
5. Update the screen automatically when new commits or new healing runs appear.

---

## 1) What This Dashboard Is Doing (Big Picture)

Think of this as a "control room" for your self-healing test framework.

Instead of opening many files manually, the dashboard gives one live screen for:

- Current git branch and recent commits
- Healing artifacts generated by test runs
- Selector memory data (what the framework has learned)
- Summary KPIs (total runs, repairs, failures, success rate)

So it is not just a pretty UI. It is a monitoring layer on top of your framework data.

## 4) Backend Architecture (`dashboard/server.js`)

The backend is intentionally simple:

1. Start a Node HTTP server on port `4000`
2. Serve `index.html` for `/`
3. Expose API routes for git/healing/memory/stats
4. Keep SSE connections open for live notifications
5. Poll every 10 seconds and broadcast updates when data changes

This keeps dependencies minimal and easy for beginners to trace.

---

## 5) API Endpoints and What They Return

The server exposes these endpoints:

- `GET /api/git`  
  Returns recent commits, current branch, changed files, diff stats, top contributors.

- `GET /api/healing`  
  Returns parsed healing runs from `test-results/healing-artifacts/*/healing-report.json`.

- `GET /api/memory`  
  Returns `test-results/selector-memory.json`.

- `GET /api/stats`  
  Returns aggregate totals like total runs, total heals, failures, estimated time saved, heal success rate.

- `GET /events`  
  SSE stream for real-time notifications (`git_update`, `healing_update`).

Beginner explanation:
"The UI does not read files directly. It asks the server, and server sends clean JSON."

---

## 6) Git Data Flow (Backend)

Three helper functions power git sections:

- `getGitLog(n)`  
  Uses `git log` to get commit hash, author, email, date, subject, refs.

- `getGitStatus()`  
  Uses `git status --short` and `git diff --stat HEAD` to show changed files and line stats.

- `getGitContributors()`  
  Uses `git shortlog -sne --all` to show top contributors.

If any command fails, code returns safe defaults instead of crashing the dashboard.

---

## 7) Healing Data Flow (Backend)

`parseHealingArtifacts()` does:

1. Read each run folder inside `test-results/healing-artifacts`
2. Open `healing-report.json`
3. Count event types:
   - `selector_recovery_applied`
   - `flow_recovery_applied`
   - `step_failed`
   - `test_completed`
4. Build a run summary object
5. Sort runs by latest start time

This makes raw event logs usable for quick UI rendering.

---

## 8) Aggregate Metrics Logic

`aggregateStats(artifacts)` calculates:

- total runs
- total selector heals
- total flow heals
- total failures
- estimated time saved  
  Formula: `selectorHeals * 0.5 + flowHeals * 1.5`
- heal success rate  
  Formula: `(selectorHeals + flowHeals) / (selectorHeals + flowHeals + failures)`

This is why KPI cards can show value without frontend doing heavy calculations.

---

## 9) Real-Time Updates with SSE

The server stores open event-stream responses in `sseClients`.

- `broadcast(event, data)` sends updates to all connected clients.
- `setInterval(poll, 10000)` checks every 10 seconds.
- If latest commit hash changes -> sends `git_update`.
- If artifact folder count changes -> sends `healing_update`.

Simple explanation for newbies:
"SSE is like keeping a phone call open from server to browser, so server can push updates immediately."

## 11) Frontend State Model

The page keeps one global state object:

```js
const S = { stats:{}, commits:[], status:{}, contributors:[], runs:[], memory:{}, stream:[] };
```

Why this helps:

- API responses are stored in one place
- render functions read from state
- easy mental model: "fetch -> save in S -> render"

---

## 12) Data Fetch Lifecycle in UI

`fetchAll()` runs 4 API calls in parallel:

- `/api/stats`
- `/api/git`
- `/api/healing`
- `/api/memory`

After each response, corresponding render functions update UI:

- `renderStats()`
- `renderGit()`
- `renderHealing()`
- `renderMemory()`
- `renderBarChart()`

It runs:

- on initial page load
- every 10 seconds
- after SSE update events

---

## 13) How Healing Runs View Works

`renderHealing()` creates run cards from `S.runs`.

When user clicks a run card:

- `selHeal(index)` is called
- It renders timeline events for that run
- Event type determines color:
  - healed (green)
  - failed/rejected (red)
  - info (cyan)

If event payload has selector details, UI shows "old selector -> healed selector".

---

## 14) How Selector Memory Table Works

`renderMemory()`:

1. Reads `S.memory` entries.
2. Splits composite context keys for readable display.
3. Prints original selector, healed selector, strategy, confidence, usage count, and last used.

This lets you explain:
"The framework is learning from past successful repairs, and we can inspect those learned entries here."

---

## 15) Chart Rendering (Canvas)

`renderBarChart()` draws a custom bar chart without external chart library.

- Takes last 14 runs
- Y value = `selectorHeals + flowHeals`
- Bar color shifts based on failure presence
- Adds lightweight grid labels and run time labels

This is a nice example of "manual canvas rendering with no dependency overhead."
